# Lania CLI v2 框架化与产品化总览

这份文档用于取代原先分散的几类文档：

- 框架定位与长期设计
- 产品作者 API 设计
- 作者侧字段迁移与 DSL 草案
- 产品化构建、打包、发布闭环
- 复杂脚手架 DSL 示例

目标不是保留所有历史讨论细节，而是给当前维护者一份“够用且收敛”的总览。

运行时模块分层、Bridge 通信、Host RPC、插件系统这些底层设计已经统一收敛到 `模块设计与通信总览.zh-CN.md`。本文只保留框架定位、作者侧 API、字段迁移和产品化闭环，不再重复展开运行时总览。

## 1. 当前定位

`lania-cli-v2` 当前已经不只是一个“内置若干命令的 CLI”，而是一个可以承载项目级命令、产品级命令和模板资产的宿主框架。

其中底层运行时由 Rust host runtime 和 Node Bridge 共同承载；本文只站在“框架层与产品层”的视角讨论作者 API、DSL 和产品化闭环。

## 2. 为什么要把“框架层”和“产品层”分开

如果继续把所有能力都堆在同一层，会很快失控：

- 宿主 runtime 会承担过多业务细节
- 产品作者会直接依赖内部实现细节
- 包结构和 API 命名会越来越不稳定
- 打包、安装态、开发态更容易出现两套行为

因此当前建议的长期边界是：

- 框架层负责稳定 runtime、统一能力和治理边界
- 产品层只负责声明命令、模板、工作流和发布元数据

## 3. 作者侧 API 应如何理解

当前仓库作者主要通过：

- `lan.config.*`
- `lania.schemas.*`

来声明项目命令、模板和扩展能力。

长期更理想的作者侧形态是一个稳定 SDK，例如：

```ts
import { defineConfig, defineSchemas } from 'lania';
```

建议作者侧 API 分两层：

### 3.1 配置入口

`defineConfig()` 负责：

- `product`
- `schema`
- `templates`
- `defaults`
- `distribution`
- `permissions`
- `interaction`
- `hookPolicy`

### 3.2 声明入口

`defineSchemas()` 负责：

- `commands`
- `presets`
- `features`
- `templates`
- `workflows`
- `dependencyRecipes`
- `mergeRules`
- `guards`
- `postActions`
- `hooks`
- `prompts`
- `conditions`
- `resolvers`

## 4. 当前字段迁移原则

为了从现有 `lan.config.* + lania.schemas.*` 平滑过渡到作者侧 SDK，建议遵循四条规则：

1. 作者已在用的高价值字段尽量平移
2. runtime 装配层字段不直接暴露，交给内部适配层消化
3. 复杂脚手架 DSL 优先进入 `defineSchemas()`
4. `lan.config` 只保留产品级、运行级、打包级配置

当前最重要的迁移判断是：

- 命令结构字段大多可以直接平移到 `commands[]`
- `plugin`、`mount`、运行时装配类字段不应继续成为作者公开 API
- prompt、workflow、模板、依赖配方应该收敛进作者 DSL，而不是继续散落在 runtime 内部细节里

## 5. 产品化闭环现在应该怎么理解

产品 CLI 不是额外的一套运行时，而是在同一宿主之上的“贴牌层”。

核心链路是：

1. 作者在 `lan.config` 中声明 `product`
2. 在 `product/lania.schemas.ts` 中声明产品命令
3. 在 `product/templates/` 中放入产品模板
4. 通过 `lan product build`
   - 生成产品 snapshot
5. 通过 `lan product pack`
   - 生成 install-root 布局
6. 通过 `lan product publish`
   - 生成 npm package、wrapper 与 publish manifest

产品安装态应保持和开发态同一套启动契约：

- Rust 仍是主入口
- Node Bridge 仍作为 runtime payload
- development mode 和 installed mode 共享 bridge 协议与装配模型

## 6. 安装与分发的稳定边界

当前建议长期保持稳定的分发边界：

- `npm wrapper`
  - 负责对外暴露最终命令名
- `Rust binary`
  - 负责命令执行与宿主治理
- `node-bridge payload`
  - 负责 JS runtime 能力
- `templates payload`
  - 负责模板资产与运行时模板目录

如果未来继续产品化，优先保持这几个边界稳定，而不是继续发明新的安装布局。

## 7. 复杂脚手架 DSL 应该长成什么样

复杂 DSL 的目标不是“炫技”，而是把真实脚手架的公共结构声明出来。

一个完整脚手架命令通常至少会包含：

- `preflight`
- `prompt`
- `preset`
- `feature`
- `renderTemplates`
- `mergeFiles`
- `writeFiles`
- `installDependencies`
- `gitInit`
- `postActions`
- `printSummary`

例如一个产品命令 `acme create-admin`，可以在一次执行中串起：

- 项目名采集
- 前端框架选择
- TypeScript / Router / ESLint / Vitest 等 feature 开关
- 模板层叠渲染
- `package.json` / `tsconfig.json` 合并
- 依赖安装
- Git 初始化
- 下一步提示

这类 DSL 当前应该服务“复杂脚手架的稳定共性”，而不是开放成无限自由的脚本语言。

## 8. 当前建议的推进顺序

如果要继续沿“框架 + 产品化”方向推进，建议优先级如下：

1. 保持宿主 runtime 和 `ctx.tools` 的稳定性
2. 收敛作者侧 SDK 入口，减少作者直接接触内部结构
3. 先打磨一个稳定的产品 CLI 样板
4. 再考虑模板市场化、更多产品矩阵和更复杂的 DSL

不建议的顺序是：

- 先开放大量高自由度 DSL
- 再回头补权限、兼容与打包契约

## 9. 当前阶段最务实的结论

`lania-cli-v2` 已经具备了框架化雏形，但当前阶段最重要的不是继续堆功能，而是：

- 保持运行时边界稳定
- 收敛作者侧 API
- 控制产品化复杂度
- 让“开发态”和“安装态”继续保持一致

如果只记一句话，可以记成：

- 先把它当成“可承载其他 CLI 的宿主框架”，再决定要不要把所有潜在能力都对外开放。
