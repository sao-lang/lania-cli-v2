## 1. 总体设计

项目当前是一个三层双运行时架构：

```text
用户命令 lan ...
-> npm wrapper
-> Rust host runtime
-> Node bridge runtime
-> 项目模板 / 动态命令 / JS 工具链 / 宿主能力
```

这三层各自的职责是：

| 层 | 目录 | 角色 | 主要职责 |
| --- | --- | --- | --- |
| 分发层 | `npm/` | 启动包装层 | 选择平台二进制、注入运行时环境变量、保持安装态与源码态兼容 |
| 控制层 | `rust/` | 宿主运行时 | 命令树、参数解析、工作流编排、统一输出、治理边界、审计 |
| 扩展层 | `ts/packages/node-bridge/` | Node 扩展运行时 | 配置加载、插件注册、模板能力、JS 生态能力、动态命令解析与执行 |

一句话概括：

- Rust 负责“命令如何被组织和治理”
- Node 负责“JS 能力如何被解析和执行”
- npm 负责“这个 CLI 如何被正确启动”

## 2. 为什么要这样分层

这套架构不是技术栈堆叠，而是有明确边界设计：

1. 命令树、统一输出、审计、权限兜底必须集中
   - 这些能力适合放在 Rust 宿主层
2. 模板、JS 工具链、项目插件、作者侧 DSL 更适合放在 Node
   - 因为它们天然依赖 JS 生态和项目本地依赖解析
3. 安装态和开发态必须共享同一套运行契约
   - 所以 `npm wrapper -> Rust binary -> Node bridge payload` 的形态被固定下来

这决定了一个核心原则：

- 命令控制权留在 Rust
- 能力扩展权开放给 Node
- 风险能力最终仍由 Rust 宿主裁决

## 3. 顶层模块图

```text
npm/cli/bin/lan.mjs
  -> rust/crates/lania-cli
     -> rust/crates/lania-host
        -> rust/crates/lania-command
        -> rust/crates/lania-workflows
        -> rust/crates/lania-plugins-command-*
        -> rust/crates/lania-node-bridge
           <stdio + jsonl>
        -> ts/packages/node-bridge
           -> core/plugin-registry
           -> plugins/*
           -> dynamic-commands/*
           -> schema-tools/*
           -> core/host-rpc
        -> ts/packages/templates
```

把它拆开看，实际上是七个核心模块群：

1. 启动与分发
2. Rust CLI 入口与命令树
3. Rust Host Runtime
4. Rust Workflows 与命令插件
5. Rust Node Bridge Client
6. Node Bridge Runtime 与插件系统
7. Template Runtime / Product Runtime / 项目扩展

## 4. 启动与分发层

### 4.1 `npm/cli/bin/lan.mjs`

关键文件：

- `npm/cli/bin/lan.mjs`

职责：

- 判断当前是源码态还是安装态
- 选择平台二进制包
- 确定 `LANIA_NODE_BRIDGE_DIR`
- 确定 `LANIA_PRODUCT_ROOT`
- 确定 `LANIA_RUNTIME_MODE`
- `spawn()` Rust 二进制并继承 stdio

它本身不参与命令解析，也不理解业务含义。它解决的是：

- 用户输入 `lan` 时应该启动哪一个二进制
- Rust 二进制应该去哪里找 Node bridge payload
- 开发态和安装态如何共用同一套启动契约

### 4.2 环境契约

启动层向下游显式传递三个关键环境变量：

- `LANIA_NODE_BRIDGE_DIR`
  - 告诉 Rust 去哪里拉起 Node bridge
- `LANIA_PRODUCT_ROOT`
  - 告诉 runtime 当前产品根目录在哪
- `LANIA_RUNTIME_MODE`
  - 区分 `development` 和 `installed`

这三个变量把“安装布局差异”隔离在了 npm wrapper 层，没有继续向 Rust 与 Node 内部扩散。

## 5. Rust CLI 入口层

### 5.1 `rust/crates/lania-cli`

关键文件：

- `rust/crates/lania-cli/src/main.rs`

职责：

- 注册内建命令插件
- 创建 `HostRuntime`
- 初始化 runtime
- 从 `clap` 解析 argv
- 还原为 `CommandContext`
- 把执行权交给 `HostRuntime::execute_command()`

这一层是 CLI 的“薄入口”，不负责承载业务逻辑。它只做三件事：

1. 把所有命令挂进同一棵树
2. 把原始 argv 解析成统一上下文
3. 把控制权移交给 host runtime

### 5.2 `rust/crates/lania-command`

职责：

- 定义命令规格
- 把 `CommandSpec` 编译为 `clap`
- 把 `clap matches` 还原为 `CommandContext`

设计意义：

- 命令定义和具体执行解耦
- 固定命令和动态命令都能落到同一套 `CommandContext`

## 6. Rust Host Runtime

### 6.1 `rust/crates/lania-host`

这是整个系统的控制面。

关键文件：

- `rust/crates/lania-host/src/runtime.rs`
- `rust/crates/lania-host/src/runtime/core.rs`
- `rust/crates/lania-host/src/runtime/lifecycle.rs`
- `rust/crates/lania-host/src/runtime/execute.rs`
- `rust/crates/lania-host/src/runtime/config/`
- `rust/crates/lania-host/src/runtime/dynamic/`
- `rust/crates/lania-host/src/runtime/host_rpc.rs`

`HostRuntime` 自身只保留三组顶层状态：

- `registries`
  - 插件、命令、handler 的注册中心
- `services`
  - logger、exec、fs、git、prompt、task、progress、package_manager、node_bridge
- `state`
  - hooks、capabilities、locale、lifecycle history

这个拆法很重要，因为它把运行时问题分成了三类：

1. “有哪些东西被注册”
2. “执行时有哪些宿主能力可用”
3. “当前运行时处于什么状态”

### 6.2 注册中心

关键文件：

- `rust/crates/lania-host/src/registry.rs`
- `rust/crates/lania-host/src/plugin.rs`

包含三类注册中心：

- `PluginRegistryImpl`
- `CommandRegistryImpl`
- `CommandHandlerRegistryImpl`

职责分工：

- `PluginRegistryImpl`
  - 注册插件
  - 解析 `before/after` 依赖
  - 调用统一 `setup_all()`
- `CommandRegistryImpl`
  - 维护命令树
  - 支持动态挂载子命令
- `CommandHandlerRegistryImpl`
  - 维护 `handler_id -> handler` 映射

它们共同解决的问题是：

- 命令如何被发现
- 命令如何被挂到树上
- 用户执行后最终应该调哪个 handler

### 6.3 生命周期

关键文件：

- `rust/crates/lania-host/src/runtime/lifecycle.rs`

生命周期主线大致是：

```text
discover
-> resolve
-> load
-> bootstrap_host_capabilities
-> plugin setup
-> runtime_start
```

作用不是“好看”，而是把这些副作用收口到统一阶段：

- 能力注册
- 插件装配
- 配置解析
- Node bridge 启动
- runtime summary 生成

这样做的价值在于：

- 新增模块时知道该接入哪个阶段
- 观测和诊断能按阶段定位问题

### 6.4 命令执行链

关键文件：

- `rust/crates/lania-host/src/runtime/execute.rs`
- `rust/crates/lania-host/src/execution/context.rs`

主链路：

```text
CommandContext
-> HostRuntime::execute_command()
-> 根据 handler_id 找 handler
-> 调 onCommandPreInit / onArgsParsed hooks
-> 加载项目配置快照
-> 构造 CommandExecutionContext
-> handler.execute(...)
-> on_success / on_error
```

这条链的设计意义是：

- 固定命令和动态命令共享同一执行主线
- hooks 可以在统一位置拦截和改写
- 统一输出、错误模型、审计都能在 host 层收口

### 6.5 配置与项目扩展引导

关键文件：

- `rust/crates/lania-host/src/runtime/config/loading.rs`
- `rust/crates/lania-host/src/runtime/config/project_extensions.rs`
- `rust/crates/lania-host/src/runtime/config/project_extensions_dynamic_commands.rs`
- `rust/crates/lania-host/src/runtime/config/project_extensions_hook_bindings.rs`

职责：

- 加载 `lan.config.*`
- 生成 runtime summary
- 解析项目扩展
- 通过 Node bridge 解析动态命令
- 把动态命令和 hook 注册回 Rust runtime

这一步是“项目配置进入宿主命令树”的关键接缝。

### 6.6 动态命令运行时

关键文件：

- `rust/crates/lania-host/src/runtime/dynamic/command.rs`
- `rust/crates/lania-host/src/runtime/dynamic/hook_registration.rs`
- `rust/crates/lania-host/src/runtime/dynamic/hook_invokers.rs`

Rust 侧的动态命令运行时不负责“解析 DSL”，只负责三件事：

1. 把 Node 返回的动态命令注册成本地命令
2. 在用户真正执行时转发到 `command.invokeDynamic`
3. 在动态 hook 触发时转发到 `hooks.invoke` 或 `hooks.invokeInline`

这说明 Dynamic Commands 的设计不是“让 Node 直接接管 CLI”，而是：

- 命令树仍由 Rust 宿主管
- 动态能力的解析和执行落在 Node

### 6.7 Host RPC

关键文件：

- `rust/crates/lania-host/src/runtime/host_rpc.rs`
- `rust/crates/lania-host/src/runtime/host_rpc/exec.rs`
- `rust/crates/lania-host/src/runtime/host_rpc/fs_domain.rs`
- `rust/crates/lania-host/src/runtime/host_rpc/git.rs`
- `rust/crates/lania-host/src/runtime/host_rpc/pm.rs`
- `rust/crates/lania-host/src/runtime/host_rpc/interaction.rs`
- `rust/crates/lania-host/src/runtime/host_rpc/log.rs`
- `rust/crates/lania-host/src/runtime/host_rpc/task_progress.rs`

职责：

- 接收 Node 发回来的 `host.*` RPC
- 提取 `__toolsPolicy`
- 路由到具体域
- 在风险域再次执行策略校验

当前域包括：

- `host.exec.*`
- `host.fs.*`
- `host.git.*`
- `host.pm.*`
- `host.log.*`
- `host.tasks.*`
- `host.progress.*`
- `host.interaction.*`

这里的关键设计原则是：

- Node 可以发起宿主能力调用
- 但是否允许，最终由 Rust 域处理器裁决

## 7. Rust Workflows 与命令插件

### 7.1 命令插件层

目录：

- `rust/crates/lania-plugins-command-*`

职责：

- 定义命令规格
- 组装命令输入 DTO
- 把命令路由到 bridge、workflow 或 local

这层是“命令入口适配层”，不是业务中台。它负责：

- `dev/build/lint/template/tools/config`
- `create/add/generate/release/sync`

### 7.2 Workflow 层

目录：

- `rust/crates/lania-workflows`

职责：

- 编排复杂命令
- 汇总宿主服务
- 在必要时调用 Node bridge 能力
- 最终收敛为 `WorkflowExecution`

它和命令插件的关系是：

- 命令插件决定“哪个 workflow 被调用”
- workflow 决定“调用顺序、状态收敛和最终输出”

### 7.3 为什么有的命令不进 workflow

并不是所有命令都需要 workflow。

分流原则大致是：

- 单次 JS 能力调用
  - 走 bridge
  - 例如 `dev`、`build`、`lint`
- 多步骤、有状态、需要汇总结果
  - 走 workflow
  - 例如 `create`、`generate module`、`release`
- 纯宿主能力
  - 走 local
  - 例如 `tools run`、`tools view`、`config`

## 8. Rust Node Bridge Client

### 8.1 `rust/crates/lania-node-bridge`

关键文件：

- `rust/crates/lania-node-bridge/src/client.rs`
- `rust/crates/lania-node-bridge/src/client/process.rs`
- `rust/crates/lania-node-bridge/src/protocol.rs`

它是 Rust 到 Node 的协议承载层，不直接理解业务。

职责：

- 拉起 Node bridge 子进程
- 维护 in-flight request 的 `pending` 表
- 区分 `event`、`response`、`host_request`
- 维护心跳与 ready 状态
- 把 `host_request` 交给宿主 handler

### 8.2 为什么单独做成一个 crate

因为它解决的是“跨进程协议”问题，不应与具体命令逻辑耦合。

这样做的好处是：

- `lania-host` 只关心“我要调一个 bridge method”
- `lania-node-bridge` 只关心“如何可靠地收发 envelope”

### 8.3 Rust 侧看到的协议形态

Rust 与 Node 之间走 `STDIO + JSONL`：

- Rust 发：
  - `handshake`
  - `request`
- Node 回：
  - `handshake`
  - `event`
  - `response`
  - `host_request`
- Rust 再回 Node：
  - `host_response`

这条链本质上是“两条方向相反的 RPC”共用一条管道：

1. Rust -> Node bridge request
2. Node -> Rust host RPC

## 9. Node Bridge Runtime

### 9.1 入口与传输层

关键文件：

- `ts/packages/node-bridge/src/index.ts`
- `ts/packages/node-bridge/src/entry/index.ts`
- `ts/packages/node-bridge/src/entry/stdio.ts`
- `ts/packages/node-bridge/src/entry/plugin-dispatch.ts`

职责分工：

- `entry/stdio.ts`
  - 进程入口
  - 处理 stdin/stdout JSONL
  - 发送 `ready` 和 `heartbeat`
  - 串行化 stdout 写入
- `entry/index.ts`
  - 组装 registry
  - 处理 handshake
  - 调 `handleExchange()`
- `plugin-dispatch.ts`
  - 把请求路由到正确插件

这里最关键的约束有两个：

1. `stdout` 只能输出协议 envelope
2. 同一个 request 的 events 必须先于 response

这两个约束保证 Rust 侧可以稳定地把 Node 看成一个“流式 RPC 服务”。

### 9.2 Plugin Registry

关键文件：

- `ts/packages/node-bridge/src/core/bridge-plugin.ts`
- `ts/packages/node-bridge/src/core/plugin-registry.ts`
- `ts/packages/node-bridge/src/core/plugin-policy.ts`
- `ts/packages/node-bridge/src/core/runtime.ts`

职责：

- 注册 builtin plugins
- 按 `cwd` 加载项目动态插件
- 对动态插件做校验与方法白名单裁剪
- 区分 `resolve()` 和 `snapshot()`
- 维护 cache，保证同一次 CLI 生命周期中插件集合稳定

这里有两个很重要的设计点：

1. 内建插件始终可用
   - 它们是 bridge 协议的一部分
2. 项目插件必须经过三重裁剪
   - 声明的 methods
   - 插件实际导出的 methods
   - policy.allowedMethods

所以 Node 插件系统不是“任意模块热插入”，而是：

- 允许项目扩展
- 但扩展范围受配置与策略共同约束

### 9.3 Builtin Plugins

当前内建插件包括：

- `config`
- `dynamic-commands`
- `lifecycle`
- `compiler`
- `product`
- `lint`
- `system`
- `template`
- `commitizen`
- `commitlint`

职责可分成四类：

1. runtime 基础能力
   - `config`
   - `lifecycle`
2. 项目级扩展能力
   - `dynamic-commands`
3. JS 工具链能力
   - `compiler`
   - `lint`
   - `product`
4. 模板和辅助能力
   - `template`
   - `system`
   - `commitizen`
   - `commitlint`

这些插件不负责协议传输，也不负责宿主治理。它们只关心：

- 给定 `method + params + context`
- 返回 `result + events`

### 9.4 Dynamic Commands

关键文件：

- `ts/packages/node-bridge/src/plugins/dynamic-commands/index.ts`
- `ts/packages/node-bridge/src/plugins/dynamic-commands/parse.ts`
- `ts/packages/node-bridge/src/plugins/dynamic-commands/execute.ts`
- `ts/packages/node-bridge/src/plugins/dynamic-commands/execute-runner.ts`
- `ts/packages/node-bridge/src/plugins/dynamic-commands/plugin-runtime.ts`

Dynamic Commands 在 Node 侧分两段：

1. `commands.resolveDynamic`
   - 解析项目声明
   - 产出 `commands + handlers + mounts + warnings`
2. `command.invokeDynamic`
   - 执行动态命令
   - 进入 local executor 或 manifest handler

这条机制把“命令的声明”和“命令的实际执行”分离开了：

- 启动时只做解析和注册
- 执行时才真正加载和运行

### 9.5 Schema Tools

关键文件：

- `ts/packages/node-bridge/src/core/schema-tools.ts`
- `ts/packages/node-bridge/src/schema-tools/`

职责：

- 为动态命令、hooks、schema handlers 注入 `ctx.tools`
- 提供统一的 JS 侧工具门面

这些工具分成两类：

1. 本地纯 JS 工具
   - path、json、text、env 等
2. 宿主代理工具
   - exec、fs、git、pm、log、tasks、progress、interaction

后一类工具并不是直接操作系统，而是通过 Host RPC 回 Rust。

这意味着：

- 对作者来说，`ctx.tools` 是统一接口
- 对 runtime 来说，底下仍然区分“本地执行”和“宿主代理”

### 9.6 Host RPC 传输层

关键文件：

- `ts/packages/node-bridge/src/core/host-rpc.ts`
- `ts/packages/node-bridge/src/entry/stdio.ts`

职责：

- 维护 Node 侧 `pending` 请求表
- 生成 `host_request`
- 等待 `host_response`
- 将 `events` 和 `result` 回填给调用者

这层是反向 RPC 的纯传输层，不应承载业务逻辑。

## 10. Template Runtime 与模板资产

### 10.1 `ts/packages/templates`

职责：

- 模板资产的结构化表达
- 模板问题与依赖元数据
- 模板渲染
- 生成文件计划

它和 `create/add/template` 的关系是：

- `template` 命令主要看模板元数据
- `create` 调模板问题、依赖、输出任务、渲染
- `add` 调增量模板渲染

### 10.2 为什么模板不直接写文件

因为模板层的职责是“声明与渲染”，不是“宿主写盘”。

因此边界被设计成：

- Node 模板层产出文件计划
- Rust workflow 决定格式化、冲突检查、hooks、写盘和安装

这样能把：

- 文件系统写入
- 冲突控制
- git 初始化
- 安装命令

统一留在宿主层治理。

## 11. Product Runtime 与作者扩展

项目当前支持三种扩展来源：

1. 内建命令插件
2. 项目动态命令
3. 项目动态插件

它们的关系容易混淆，实际边界如下：

### 11.1 命令插件

- Rust 侧注册
- 固定随 CLI 发布
- 负责宿主级命令

### 11.2 动态命令

- 由项目配置声明
- 启动时解析后挂进 Rust 命令树
- 执行时由 Node 处理

### 11.3 项目动态插件

- 由 `lan.config.plugins` 声明
- 通过 Node plugin registry 按 `cwd` 加载
- 为 bridge 提供额外 method

两者区别是：

- 动态命令扩展的是“命令面”
- 项目插件扩展的是“bridge 方法面”

它们经常一起出现，但不是一回事。

## 12. 模块之间的通信模型

### 12.1 启动通信

```text
npm wrapper
-> env vars
-> Rust main
-> HostRuntime::initialize()
```

这里通信主要靠：

- 进程 argv
- 环境变量

### 12.2 命令执行通信

```text
clap matches
-> CommandContext
-> CommandExecutionContext
-> handler.execute()
```

这里通信主要靠 Rust 内部 DTO 和 trait。

### 12.3 Rust -> Node 通信

```text
BridgeRequest
-> stdio/jsonl
-> handleExchange()
-> plugin.handle()
-> BridgeResponse + BridgeEvent[]
```

这里通信主要靠：

- method 字符串
- JSON params/result
- 事件流

### 12.4 Node -> Rust 通信

```text
hostCall()
-> host_request
-> HostRpcAdapter
-> host.* domain
-> host_response
```

这里通信主要靠：

- `host.xxx.*` method namespace
- `__toolsPolicy`
- JSON payload/result

### 12.5 Workflow 内部通信

workflow 之间不直接互调协议，而是共享：

- `WorkflowServices`
- `WorkflowExecution`
- 统一的宿主服务和 bridge 能力

这让 `create/release/generate` 这些复杂命令虽然逻辑不同，但能共享同一套运行骨架。

## 13. 通信边界与治理点

最重要的治理点有四个：

### 13.1 命令治理点

位置：

- `HostRuntime::execute_command()`

职责：

- 统一进入执行链
- 做 hooks、上下文和错误收敛

### 13.2 插件治理点

位置：

- `core/plugin-registry.ts`
- `core/plugin-policy.ts`

职责：

- 限制项目插件来源和 methods
- 稳定 `cwd` 生命周期内的插件集合

### 13.3 宿主能力治理点

位置：

- `runtime/host_rpc.rs`
- `host_rpc/*`

职责：

- 在 Rust 侧再次做策略兜底
- 把风险能力限制在可审计边界内

### 13.4 文件写入治理点

位置：

- Rust workflows
- `write_files_with_hooks(...)`
- `apply_contract_generation(...)`

职责：

- 模板层和 schema 层只产出计划
- 真正写盘必须回到宿主

## 14. 维护时最该守住的边界

如果后续继续扩展，这些边界最好不要被破坏：

1. `npm wrapper` 不承载业务逻辑
2. Rust 继续保持命令树主控权
3. Node bridge 继续保持“插件运行容器”角色
4. 动态命令和项目插件不要混成一个概念
5. Host RPC 继续是 Node 使用宿主能力的唯一正式入口
6. 文件写入与高风险 exec 继续由 Rust 宿主最终裁决

如果这些边界还在，仓库即使继续长，也不会失去结构。

## 15. 推荐阅读顺序

如果你要系统吃透这套设计，建议按这个顺序：

1. `npm/cli/bin/lan.mjs`
2. `rust/crates/lania-cli/src/main.rs`
3. `rust/crates/lania-host/src/runtime.rs`
4. `rust/crates/lania-host/src/runtime/execute.rs`
5. `rust/crates/lania-node-bridge/src/client/process.rs`
6. `ts/packages/node-bridge/src/entry/stdio.ts`
7. `ts/packages/node-bridge/src/core/plugin-registry.ts`
8. `ts/packages/node-bridge/src/plugins/dynamic-commands/`
9. `ts/packages/node-bridge/src/core/host-rpc.ts`
10. `rust/crates/lania-host/src/runtime/host_rpc.rs`
11. `rust/crates/lania-workflows/src/`
12. `ts/packages/templates/src/`

读完这 12 个入口，再回头看命令级链路，会轻松很多。
