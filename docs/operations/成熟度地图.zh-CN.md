# Lania CLI v2 成熟度地图

这份文档基于当前仓库已经通过的自动化验证结果，回答一个更实际的问题：

- 哪些能力已经被验证过，可0以较放心地继续使用
- 哪些能力虽然能用，但仍应谨慎评估
- 哪些能力还不适合当成当前阶段的核心押注

本文不是从“设计愿景”出发，而是从“现有测试与回归结果”倒推当前成熟度。

## 2. 分级标准

为了让判断更可执行，本文使用三档成熟度：

### `我现在敢用`

满足以下多数条件：

- 有专门的自动化测试覆盖
- 不只测单个函数，还测到了完整链路或工作流
- 本轮回归中已稳定通过
- 对当前 CLI 核心价值直接相关

### `我谨慎用`

满足以下特征之一：

- 已有测试，但跨层复杂度高
- 能力边界比较宽，组合空间大
- 当前看起来可用，但后续仍容易被配置、插件或平台差异放大风险

### `我先别押宝`

满足以下特征之一：

- 当前验证偏局部或偏冒烟
- 更像框架扩展能力，不是当前主业务闭环
- 一旦出问题，定位与维护成本明显高于收益

## 3. 我现在敢用

这一层的特点是：已经被多层验证覆盖，而且是当前仓库最核心的用户路径。

### 3.1 CLI 主命令骨架

当前 CLI 主入口已经过了完整 Rust workspace 回归，并且有单独的 `phase3_e2e` 入口覆盖：

- `help_and_config`
- `build_and_lint`
- `create_and_add`
- `template_commands`
- `generate_workflows`
- `sync_and_release`
- `distribution_and_dynamic`

这说明以下能力不再只是“目录里有代码”，而是经过真实命令路径验证：

- 命令注册与解析
- `help` / `config`
- 构建与 lint 命令接线
- create/add/template/generate/sync/release 主命令分发

判断：`我现在敢用`

原因：

- 有 CLI e2e
- 有 Rust workspace 全量测试
- 是当前仓库最核心的使用入口

### 3.2 `create` / `add` 工作流

`create` 相关工作流测试覆盖较完整，至少包括：

- 写出模板文件
- 使用 bridge template
- 发现 bridge-only template
- preview 模式不落盘
- 初始化 git 仓库
- 非空目录校验
- 当前目录创建

对应测试位于：

- `rust/crates/lania-workflows/src/tests/create_add.rs`

这意味着：

- 脚手架生成不是只靠静态模板存在
- 模板发现、模板写出、目录策略、git 初始化都已经过验证

判断：`我现在敢用`

### 3.3 `generate module` / `generate api`

原因：

### 3.4 `sync` / `release` 工作流

### 3.5 任务、交互、进度这些宿主底层能力

- 本轮最关键的取消回滚卡住问题也已修复并回归通过

## 4. 我谨慎用

这一层的特点是：已经被验证过，但复杂度高、边界宽、后续维护风险仍然显著。

### 4.1 Node Bridge

`node-bridge` 的测试面其实很广，覆盖了：

- runtime
- plugin policy
- entry
- dynamic commands
- workflow steps
- transaction engine
- guard engine
- merge engine
- system
- template
- compiler
- lifecycle
- product
- contract
- commit tools

对应测试主要位于：

- `ts/packages/node-bridge/src/core/`
- `ts/packages/node-bridge/src/plugins/`
- `ts/packages/node-bridge/src/entry/`

判断：`我谨慎用`

不是因为它没测，而是因为：

- 它横跨 Rust 宿主、Node 运行时、配置、插件和项目上下文
- 任何一层变化都可能引出跨边界问题
- 组合空间明显大于普通模块

更准确地说，`node-bridge` 已经不是“实验草稿”，但也还没到“可以完全忘掉复杂度”的程度。

### 4.2 Dynamic Commands

Dynamic Commands 当前已经被多处验证：

- TS 侧有独立测试
- CLI e2e 里有 `distribution_and_dynamic`
- 宿主 runtime 也覆盖了 dynamic bootstrap 与可见性

判断：`我谨慎用`

原因：

- schema、prompt、handler、plugin、tools、host RPC 一旦自由组合，状态空间非常大
- 测试证明它“能工作”，但不代表所有组合方式都已经被充分锤过

如果当前团队要使用它，建议：

- 只开放少量稳定 schema 能力
- 尽量减少自由组合和复杂 prompt 流程
- 把它当作“受控扩展能力”，而不是“无限自由的框架 DSL”

### 4.3 宿主 Runtime / 插件机制

`lania-host` 当前通过的测试已经证明：

- runtime 能装配
- project plugin 能发现和加载
- installed product root commands 可见
- host RPC 主链能工作

判断：`我谨慎用`

原因：

- 这层是整个系统最重的基础设施层
- 一旦继续叠新 feature，很容易重新变得难验证
- 它的复杂度不是来自实现质量差，而是来自它本身承担了太多框架职责

### 4.4 模板系统

模板系统当前具备两层验证：

- TS 模板包自身测试
- Rust `create` 工作流对模板写出与目录结果的验证

判断：`我谨慎用`

原因：

- 模板种类和模板变量一旦继续增长，维护成本会很快上升
- 模板天生更容易被“边缘组合场景”打穿

所以模板系统现在是“可用”，但建议继续控制模板数量和分叉复杂度。

### 4.5 `product` 贴牌 CLI 能力

这块能力不是没验证：

- `node-bridge` 有 product 相关测试
- CLI e2e 有 distribution / dynamic 模块
- 打包和 Verdaccio 发布演练也都跑通过

判断：`我谨慎用`

原因：

- 这是一个天然跨层的能力：配置、构建资产、bridge、wrapper、分发都要协作
- 它属于“能力很强，但维护成本也高”的部分

如果当前目标是先把主 CLI 做稳，`product` 更适合作为增强项，而不是第一优先级主战场。

## 5. 我先别押宝

这一层不代表“完全不能用”，而是代表“当前阶段不适合当成核心战略依赖”。

### 5.1 Dynamic Commands 的复杂组合玩法

例如：

- 很深的 prompt 流程
- 大量条件跳转
- 多阶段事务与合并规则
- 高度自定义的插件 handler

这些能力当前并非完全没验证，但不适合作为当前阶段的主要生产依赖。

判断：`我先别押宝`

原因：

- 它们往往最能展示框架能力
- 但也最容易造成理解成本和维护成本同时上升

### 5.2 Product CLI 的大规模铺开

虽然贴牌 CLI 已经具备了基础闭环，但如果当前团队还在收敛主 runtime，本阶段不建议把多个产品线都压到这条能力上。

判断：`我先别押宝`

更实际的策略是：

- 先让一个产品样板稳定
- 再决定是否把它推广成标准能力

### 5.3 所有“看起来很强”的框架扩展点

如果一个能力同时满足下面两条，就应该暂时克制：

- 它不是当前最重要的用户路径
- 它需要跨 Rust / Node / 配置 / 分发 多层协作

这类能力最容易在“设计上很优雅”和“实际维护很痛苦”之间失衡。

## 7. 最实际的使用建议

如果现在要基于这套系统继续推进，我建议采用下面的策略：

### 优先继续投入

- `create`
- `generate`
- `sync`
- `release`
- CLI 主命令主链
- task / prompt / progress 等宿主底层稳定性

### 控制增长速度

- `node-bridge`
- 模板系统
- 动态命令基础能力
- product CLI

### 暂时避免继续扩张

- 复杂 schema DSL
- 过深的 dynamic prompt flow
- 多层 product 化矩阵
- 不直接服务当前主路径的“框架展示型能力”

## 8. 一句话结论

当前仓库已经可以从“很多功能像没验证过”更新为：

- 核心 CLI
- 主要工作流
- 宿主底层
- Node Bridge 主能力

都已经被真实测试覆盖并通过回归。

真正需要继续克制和收敛的，不是主链路，而是那些高抽象、强扩展、跨多层的框架化能力。
