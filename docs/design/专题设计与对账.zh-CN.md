# Lania CLI v2 专题设计与对账

这份文档收敛原先分散的专题文档，重点保留“当前仍然有维护价值”的结论，包括：

- `v2.3` `ctx.tools` 与 host-backed tools
- 模板边界、模板版本与迁移层次
- 插件安全策略
- 类型契约与兼容层
- Phase 3 工程化增强的实际收敛结果

## 1. `ctx.tools` 与 host-backed tools

`v2.3` 的核心结论已经稳定：

- 动态命令和 inline hooks 都可以使用 `ctx.tools`
- Node 侧通过 `tools.host.call()` 安全回调 Rust 宿主能力
- 核心工具域已覆盖：
  - `git`
  - `pm`
  - `exec`
  - `config`
  - `fs`
  - `log`
  - `text`
  - `tasks`
  - `progress`
  - `interaction`
  - `bridge`
- 高频增强工具已覆盖：
  - `path`
  - `workspace`
  - `env`
  - `json`
  - `result`

对维护者来说，最重要的不是记住全部 surface，而是记住边界：

- JS 侧编排通过 `ctx.tools`
- 高风险宿主能力仍由 Rust 控制和审计

## 2. 模板边界与模板运行时

模板体系的当前边界已经比较清楚：

### 2.1 Rust 拥有的部分

Rust workflow 层负责：

- 内建 declarative template inventory
- prompt 默认值
- 包管理器规划
- 文件冲突策略

内建模板当前包括：

- `spa-react`
- `spa-vue`
- `toolkit`

### 2.2 Node 拥有的部分

Node template runtime 负责：

- 动态模板执行
- EJS 渲染
- `questions.ts`
- `dependencies.ts`
- `config.ts`
- 第三方模板扩展目录

### 2.3 当前规则

- 如果模板属于 Rust 内建 inventory，Rust 语义优先
- 如果模板不属于 Rust 内建清单，则回落到 Node runtime 执行
- 内建模板的 source-of-truth 仍在 Rust，Node 更像兼容层和扩展层

### 2.4 当前版本约定

- 当前 template schema version 为 `1`
- metadata 允许在 `v1` 内做增量扩展
- 如果发生字段破坏性改名、ownership contract 改变或结果形态变化，应提升 schema version

## 3. 插件安全与信任模型

插件安全策略当前可以压缩为三层：

### 3.1 来源约束

- 支持 `package` 和 `local_path`
- 本地插件必须位于 workspace root 内
- 本地插件文件扩展名受限

### 3.2 包信任约束

- 一方插件可按前缀策略直接通过
- 三方插件需要 allowlist
- 开启签名要求时，需要声明 signature 或进入签名 allowlist

### 3.3 方法暴露约束

- 插件模块必须显式声明可暴露的方法
- 最终暴露方法会与 allowlist 做交集
- 插件运行时错误会被转换成标准 bridge error，而不是直接击穿 bridge 进程

## 4. 类型契约与兼容层

当前 v2 的公共契约已经从“一个巨大的 shared TS 类型包”转成两层：

### 4.1 Rust crate 直接导出的公共契约

主要包括：

- `lania-command`
- `lania-host`
- `lania-config`
- `lania-task`
- `lania-progress`
- `lania-prompt`
- `lania-fs`
- `lania-pm`
- `lania-git`
- `lania-presentation`

结论：

- 命令模型、宿主抽象和服务层类型按能力拆分维护
- 不再继续维持一个巨大的 `shared` 类型入口

### 4.2 通过 bridge 协议暴露的公共契约

主要包括：

- `HandshakeRequest`
- `HandshakeResponse`
- `BridgeRequest`
- `BridgeResponse`
- `BridgeError`
- `BridgeEvent`
- `BridgeExchange`

结论：

- 跨运行时公共契约优先由协议定义，而不是由 Node 内部 TS 类型直接对外暴露

## 5. Phase 3 工程化增强

“Phase 3 工程化增强”的真实完成范围，应理解为脚手架执行内核增强，而不是产品打包阶段。

当前已经形成闭环的能力：

- 统一 `ctx.scaffold` 计划入口
- 标准化 workflow steps 执行链
- merge engine runtime
- 最小 transaction / rollback 模型
- preflight / guards
- `printSummary` / `postActions` UX
- transaction / compensation 作者侧暴露

当前仍应保守对待的边界：

- 依赖安装的真正自动回滚
- Git 初始化与首个提交的真正自动回滚
- 更丰富的 host-facing render model
- 更大规模的产品打包分发矩阵
