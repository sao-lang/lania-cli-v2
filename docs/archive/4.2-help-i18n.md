# 4.2 Help, Copy, And I18n Strategy

## Current Baseline

v2 currently keeps CLI help, option descriptions, workflow notes, and JSON error messages in English.

That is intentional for the current release baseline because:

- command specs are generated from Rust plugin metadata
- host/runtime outputs are already structured and machine-readable
- user-visible strings are still being consolidated behind presentation and logging layers

## Long-Term Rule

String ownership is split by layer:

- command help and option copy live in Rust command specs
- structured runtime errors live in Rust/Node error codes plus stable JSON fields
- styled terminal rendering and future localized text live in `lania-presentation`

This means localization should happen at the presentation boundary, not by scattering translated strings through workflow logic.

## Legacy Locale Reconciliation

The legacy reference file `packages/core/locales/zh.json` in the old repository is currently empty, so there is no residual locale asset that must be migrated verbatim into v2.

For v2 launch, the retained user-facing compatibility target is therefore:

- preserve command names and option semantics
- preserve stable machine-readable error/result fields
- preserve workflow note intent where old behavior surfaced actionable guidance

## Release Policy

Before introducing full i18n:

- keep new help/error strings concise and structured
- prefer stable error codes over prose-only diagnostics
- avoid baking localized copy into protocol payload schemas

When i18n is added:

- `lania-presentation` becomes the translation boundary
- CLI help, logger copy, and renderer output share the same locale source
- protocol payloads remain locale-neutral unless a field is explicitly display-only
